# FSRS Profiles UX/UI Design Specification

## 设计理念

### 核心原则
1. **渐进式披露** - 80%用户只看"策略"，专家才看 weights
2. **强保护** - Append-only 版本控制 + 回滚点 + 应用确认
3. **可视化结果** - 用户关心"改了之后会怎样"而不是"参数是什么"
4. **数据完整性** - 校验函数 + RPC 封装 + 缓存失效机制

### 技术实现要点（重要）

#### Append-Only 版本控制
- **每次保存 = INSERT 新行**（version+1）
- **旧版本 is_active=false**（归档保留）
- **禁止 UPDATE 核心字段**（触发器保护）
- **天然支持回滚 + 审计**

#### 缓存失效策略
- Simulation 缓存绑定 `cards_snapshot_revision`
- Card 变化时自动 +1（通过触发器）
- 读取缓存时检查 revision 是否一致

#### Fallback 链
```
subject profile → global profile → preset default
```
确保用户始终有有效配置，即使从未设置过。

### 用户分层
| 用户类型 | 比例 | 需求 | 暴露深度 |
|---------|------|------|---------|
| 普通用户 | 80% | 自动优化 + 工作量可控 | Overview + Presets |
| 进阶用户 | 15% | 选择策略风格 + 学科覆盖 | Tuning 控件 |
| 专家用户 | 5% | 导入导出 + 精细调参 | Advanced weights |

## 技术实现注意

### Simulation 计算
- 需要访问用户的 `cards` 表
- 计算每张卡的下次 due 时间
- 汇总为每日 due 数量

### 权重校验
- 检查数组长度（FSRS-5 = 21）
- 检查数值范围
- 检查 schema version 兼容性

### 缓存策略
- Simulation 结果缓存 1 小时
- Profile 列表使用 React Query 缓存
- 权重变更时失效相关缓存
